# AIMORI 深層学習ロードマップ

> このドキュメントは、AIMORI の知財検知エンジンに深層学習（DL）をどこまで・どの順序で内製していくかを整理したものです。
> 対象読者: プロダクト/技術の意思決定者。コードではなく「投資の順序と判断基準」を扱います。

## 0. 現状（フェーズ0）の正確な姿

現行 PoC は **Google Vision API `WEB_DETECTION`（逆画像検索）を呼び出すだけ** の薄いラッパーです。

- 自前の DL・埋め込み（embedding）・知覚ハッシュ・ベクタDB は **一切ない**（依存は `Flask` のみ）。
- **数値の類似度スコアは存在しない**。Vision が返すバケット（`fullMatchingImages` / `partialMatchingImages` / `visuallySimilarImages`）を、カテゴリ順（完全一致→部分一致→掲載→類似）で並べているだけ（`poc/reverse_search.py:240-256`、`_SEVERITY_ORDER`）。
- 「侵害疑い」の判定は、ヒット URL のホスト名が `TARGET_PLATFORMS`（メルカリ/minne/Creema/pixiv 等）の許可リストに載っているか、で決まる（`poc/reverse_search.py:76-86`）。

**つまり現状の AIMORI は「自分の判断軸」を1つも持っていません。** 検知の質は完全に Google のインデックスと分類に依存します。

### 現状の2つの構造的弱点

1. **許可リスト依存**: `TARGET_PLATFORMS` に載っていない新規マーケットプレイスのヒットは、黙って「参考（その他ページ）」に降格される。監視対象の拡張が手作業。
2. **リコールが Google インデックス頼み**: 未インデックスの出品や、加工された（色変え・反転・コラージュ）リポストは原理的に拾えない。UI 自身がこの限界を注記している。

## 1. DL が絡む3つのレイヤと、価値が出る場所

知財類似検知に DL を入れるとき、関与する領域は大きく3層に分かれます。**投資対効果の順序を間違えないこと**が重要です。

### レイヤ1: 特徴抽出・類似度（本丸）
- 意匠/商標/イラストの「見た目の類似」を、画像→ベクトルに変換して近傍探索する。
- 出発点は **既製の埋め込みモデル**（CLIP / DINOv2）。ドメイン特化が必要になったら **メトリックラーニング**（ArcFace / triplet loss）でファインチューニング。
- ここが「モデル設計・チューニング」が最も価値を生む場所。**ただし着手は後述の通り最後**。

### レイヤ2: 評価設計（実際のボトルネック）
- 知財の類似は「正解ラベル」が主観的・法的判断寄りで、単純な accuracy では測れない。
- **Recall@k を主指標**にする（＝侵害の見逃しを最小化）。mAP は補助。
- 「見逃しゼロ寄り」に振った運用点（PR 曲線のどこで閾値を切るか）が製品品質を決める。最終判断は弁理士/法務が行う前提で、AI 側は高リコールに寄せる。
- **DL のモデルより先に、この評価基盤を持つことが投資効率を最大化する。** 評価できないと、以降のあらゆる改善が勘になる。

### レイヤ3: アルゴリズム設計（スケール時）
- 監視対象が増えると総当たり比較が破綻する。**ANN（HNSW / IVF-PQ、Qdrant）** でのインデクシング設計が効いてくる。
- これはユーザー/コーパスが一定規模に育ってからの課題。

## 2. フェーズ対応表（内製 DL の投資深度）

| フェーズ | やること | DL の関与 |
|---|---|---|
| **0（現状）** | Vision 逆画像検索プロキシ | 閾値・PF 許可リスト・前処理のみ（学習・推論なし） |
| **1** | MVP ＋ **評価データ整備・指標設計** | 評価設計が主役。モデルはまだ作らない |
| **2** | 既製埋め込み/pHash による **二次リランキング** | 推論のみ（学習なし）。Vision 候補を自前スコアで並べ替え |
| **3** | ドメイン特化ファインチューニング ＋ **CLIP/DINOv2 + Qdrant** | 学習・ANN 設計が本格化 |

**設計思想**: 各フェーズは前フェーズが生むデータを燃料にする。評価基盤（フェーズ1）→ ラベル済みデータ → リランキング検証（フェーズ2）→ ユーザーフィードバック → 大規模化・特化学習（フェーズ3）、という順にデータの飛輪（フライホイール）を立ち上げる。

## 3. build-vs-buy と「内製学習を始めるタイミング」

### 原則: 自前学習は「既製で頭打ちの証拠」が出てから
- ラベル付きデータが薄い段階でモデルを学習しても過学習するだけ。
- 既製埋め込み（CLIP / DINOv2）＋ ベクタ検索 ＋ 閾値チューニングだけで、「効く/効かない」の判断は十分にできる。
- **内製モデル自作は、既製手法の性能上限に当たったことを評価基盤が示してから投資する。**

### 既製手法で回収できる範囲（フェーズ2で先に取る）
- 色変え・反転・軽微な加工・切り抜きの **near-duplicate（近重複）検知** は、まず **知覚ハッシュ（pHash）** で、次に **CLIP/DINOv2 の埋め込み** で回収できる。ここは学習不要。
- 現状の「加工リポストを拾えない」弱点の相当部分は、フェーズ2の既製手法で埋まる見込み。

### 内製学習に踏み込むべきサイン（フェーズ3）
- 既製埋め込みでは「イラスト/ロゴ/パッケージ特有の類似」（例: 構図は同じだが画風を変えたパクリ）を Recall@k で十分に拾えない、と評価基盤が示したとき。
- このとき初めて ArcFace 等のメトリックラーニングで知財特化し、**競合が真似できない検知精度**を参入障壁にする。

## 4. デプロイ制約: なぜ ML は今の Vercel 関数に載せないか

現行のスキャン処理は Vercel の Python サーバーレス関数で動いている。ここに CLIP 等の ML 推論は載せられない:

- **書き込みは `/tmp` のみ・かつ揮発性**（モデル重み/インデックスを永続化できない）
- **レスポンス上限 ~4.5MB**、**アップロード上限 14MB**
- **実行時間制限**（Hobby ~10s / Pro ~60s、最大 300s）＋ **常駐プロセスなし・コールドスタート**（数百MBのモデルをロードするには不向き）
- **GPU なし**

→ 実運用の ML 推論は **別の常駐コンテナ/サービス（GCP 等）** に置く。SPEC も「Vercel（フロント）＋ コンテナ（バックエンド）」の分離を想定している。フェーズ2の効果検証（本リポジトリの `poc/rerank.py` / `poc/eval_rerank.py`）は **完全にオフライン（ローカル実行）** で行い、Vercel 経路には触れない。

## 6. 次の一歩

本リポジトリには、フェーズ2の効果を**オフラインで最小コスト検証する**ハーネスを同梱している:

- `poc/rerank.py` — Vision が返した候補画像に対し、pHash → （必要なら）CLIP/DINOv2 で **本物の 0–1 類似度スコア**を付与し再ランキングする。
- `poc/eval_rerank.py` — ラベル付きセットに対し Recall@k / mAP を算出し、「Vision 順」vs「pHash 順」vs「CLIP 順」を比較する。

これは飛輪の**点火装置**であって、エンジン本体（常駐バックエンド＋ユーザーフィードバック導線）ではない。効果が数値で確認できてから、フェーズ2の実インフラ投資に進む。使い方は `poc/README.md` の「オフライン評価ハーネス」節を参照。
