## 3. MVP機能スコープ

### 含む機能
1. **作品登録** — 画像アップロード（〜数十点）
2. **定期スキャン・類似検知** — Google Vision API Web Detection を使った逆画像検索。転載（同一画像）・模倣品（類似画像）・デザインパクリ（全サイト類似）の3種を検知
3. **アラート** — ヒットURL・掲載サイト名・一致種別（転載/模倣品候補/デザインパクリ）・取得日時を通知
4. **案件管理** — 発見 → 通報済み → 対応中 → 解決 のステータス管理

### 含まない機能（MVP外）
- 削除要請テンプレート自動生成（非弁リスクの法的確認完了後に追加）
- 回収代行・法的手続き代行（弁護士費用・非弁リスクのため）
- 英語・中国語テンプレ（フェーズ2以降）
- 海外EC（SHEIN/Temu）対応（フェーズ3以降）

## 4. 技術スタック

### 検知エンジン（確定）

**Google Vision API Web Detection（逆画像検索代行モデル）**

```
クリエイター → AIMORI → Google Vision API → Googleの全Webインデックスを検索
                                           → メルカリ/minne/Creema等のURLがヒット
                                           → アラート
```

| 検知種別 | 使用フィールド | 対象 |
|---------|-------------|------|
| 転載・転売 | `pagesWithMatchingImages` / `fullMatchingImages` | 同一・ほぼ同一の画像が載っているページ |
| **模倣品** | `visuallySimilarImages` | 別画像だが視覚的に類似するもの（＝模倣品候補） |
| **デザインパクリ** | `visuallySimilarImages`（全件） | ターゲットPF問わず全サイトの類似画像（`--all` モード） |

**なぜこのモデルか（自前クロールを採用しない理由）**
- 自前クロール = 各社利用規約違反リスク（高）+ 業務妨害リスク（高）
- Google Vision API = Googleが合法的にクロール済みの索引を借りるだけ。法的リスクがほぼゼロ
- 原価：**月1,000ユニットまで無料、超過分 $3.50/1,000ユニット**（約100円/ユーザー/月）

**料金（2026-07時点、出典: cloud.google.com/vision/pricing）**

| 月間利用量 | 料金 |
|-----------|------|
| 〜1,000ユニット | **無料** |
| 1,001〜5,000,000ユニット | $3.50 / 1,000ユニット |

## 7. 法務リスクと対策

### リスク1：クローリングの適法性 → **Option C（逆引き代行モデル）により解消**

| アプローチ | 法的リスク |
|-----------|----------|
| 自前クロール（旧計画） | 高（利用規約違反・業務妨害） |
| **Google Vision API 代行（現計画）** | **ほぼなし**（Googleの合法的索引を利用） |

- 著作権法30条の4（情報解析目的の複製）は自前クロール時の論点。現モデルでは不要。
- 業務妨害（Librahack事件リスク）も現モデルでは発生しない。
- ただし弁護士スポット相談は、ユーザーへの画像表示方法（著作権法47条の5）の確認のために引き続き推奨。

### リスク2：非弁行為
- **境界線**: テンプレート提供 = OK / 代理送付・交渉 = NG
- MVPでは削除要請テンプレートを除外（法的確認完了後に追加）

## 9. GTM（Go-To-Market）戦略

### 獲得チャネル
- pixiv・X上のイラストレーターコミュニティへの直接アプローチ
- minne/Creema作家コミュニティ、ハンドメイド系ブロガー/YouTuber
- 被害体験を発信している作家をアーリーアダプター兼広報に

### 無料フック
「無料スキャン体験」— 作品画像を受け取り、運営者が手動でチェックして被害レポートを返す（MVP実装前はコンシェルジュ型）。PoCスクリプトを使えばすぐ実施可能。

**フェーズ0の運用フロー**: LPのCTA（`contact@flowmate.jp`宛mailto）から問い合わせ → 運営者が`WEBAPP_USER`/`WEBAPP_PASS`（Basic Auth、`poc/webapp.py`）を個別にメールで案内 → 申込者が`/poc`の内部ツールに直接アクセスして自分で画像をアップロード・確認、または運営者が代行して結果を送る。現状の認証はアプリ全体共通の1組のBasic Authであり、申込者ごとに個別のユーザーIDを発行する仕組みではない点に注意（少人数運用の間はこれで割り切る前提。個別発行・失効が必要になった場合は別途実装が要る）。

### PR文脈
SHEIN/Temu模倣問題は報道フックが強い。初期はプレスリリース不要、作家コミュニティでの口コミを優先。
